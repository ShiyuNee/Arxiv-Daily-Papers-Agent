# papers.cool 论文整理（按 Topic 分类版）

## 概述

使用 Python 爬虫脚本 `papers_cool_crawler.py` 爬取 papers.cool 页面并保存原始 JSON，然后**基于 abstract 对论文按 topic 分类**，为每个 topic 撰写研究动态总结，**直接读完所有论文 abstract 后为每篇撰写高质量中文 TL;DR**（不翻译完整摘要），按排除条件过滤后保存。

**目录和文件命名规范（严格遵守！）：**

所有文件保存在项目根目录下，按 `{category}_{date}` 建文件夹，总结文件固定命名为 `summary.md`：

```
项目根目录/
├── {category}_{date}/           # 例如 cs.CL_2026-05-14/
│   ├── summary.md               # 最终整理报告（固定文件名，不可更改）
│   ├── papers_{category}_{date}.json   # 爬虫输出的原始 JSON
│   └── papers_{category}_{date}.md     # 爬虫输出的原始 Markdown
```

**示例：** 处理 cs.CL, 2026-05-14 的论文时：
- 文件夹：`cs.CL_2026-05-14/`（在项目根目录下新建）
- 爬虫输出：`cs.CL_2026-05-14/papers_cs.CL_2026-05-14.json`、`cs.CL_2026-05-14/papers_cs.CL_2026-05-14.md`
- 最终报告：`cs.CL_2026-05-14/summary.md`

**禁止事项：**
- ❌ 不要把文件保存到 `~/Desktop/daily_papers/` 或其他任意位置
- ❌ 不要把总结文件命名为 `{category}-papers-{date}.md` 或其他名称，必须统一为 `summary.md`
- ❌ 不要在文件夹外创建任何输出文件

**核心原则：只写 TL;DR，不翻译摘要。读完所有 abstract 后直接写 TL;DR 写在最终输出文件中，不要写到别的文件里。**

主要目标：
- 稳定爬取指定 arxiv 类别所有论文
- **基于 abstract 分类到 topic**
- **按 topic 归纳当日研究动态**
- **为每篇论文撰写中文 TL;DR**（问题 + 方法，两句话）
- 按条件排除特定类型论文
- 保存到结构化目录
- 原始数据可追溯、可重跑

## 参数提取

从用户请求中提取以下参数：

| 参数 | 类型 | 必填 | 默认值 | 说明 |
|------|------|------|--------|------|
| `url` | string | ✅ | - | papers.cool 完整 URL |
| `category` | string | ✅ | - | arxiv 类别，如 cs.CL, cs.IR |
| `date` | string | ✅ | - | 日期，如 2026-04-08 |
| `show` | int | ❌ | 300 | 页面展示论文数目上限 |

**典型触发语句：**
```
整理 https://papers.cool/arxiv/cs.CL?date=2026-05-11&show=300
把 cs.CL 5 月 11 日的论文整理一下
```

**参数提取规则：**
- 从 URL 提取 `category`（如 `cs.CL`）
- 从 URL 提取 `date`（如 `date=2026-05-11`）
- 从 URL 提取 `show`（如 `show=150`），如无则默认 150
- 如无 URL，从文本提取类别和日期

## Topic 候选类别体系

Topic 候选列表维护在**独立文件**中：

**文件路径：** 项目根目录下的 `TOPICS.md`

模型在分类时**必须先读取 TOPICS.md**获取最新的 topic 列表，分类完成后如有新增 topic，**必须同步更新 TOPICS.md**。

### Topic 分类规则

1. **先读取 TOPICS.md**：每次分类前，读取 TOPICS.md 获取当前完整的 topic 候选列表
2. **优先使用已有 topic**：分类时首先检查论文 abstract 是否匹配某个已有 topic
3. **允许跨 topic**：一篇论文可以归属多个 topic，取**最核心的一个**作为主分类
4. **自动扩充**：当论文无法归入任何现有 topic 时：
   - 创建新 topic，分配新 Topic ID（小写英文短横线格式，如 `skill-learning`）
   - 写明 Topic 名称和说明
   - **在 TOPICS.md 的「🆕 新增 Topic 记录」区域追加新条目**，标注新增日期和当日论文数
   - **在输出 markdown 的「🆕 新增 Topic」区域记录**
5. **最小粒度**：topic 不宜过细，某 topic 当日只有 1 篇论文时，考虑并入相关 topic

## TL;DR 撰写规范（最核心！整个流程最易出错的环节）

### 为什么要 TL;DR 而不是完整翻译摘要

**绝对不要翻译完整摘要！** 原因：
- 翻译全文太慢，200+ 篇论文一个一个翻译根本做不完
- 用户可以直接看英文 abstract，翻译摘要信息冗余
- TL;DR 两句话说清问题和方法，阅读效率高得多

### TL;DR 格式

固定格式，**每个 TL;DR 都必须严格遵循**：

```
TL;DR: 问题：[具体问题+为什么现有方法不行] 方法：[方法名+具体怎么做的+关键结果（如有）]
```

### TL;DR 质量标准

**核心标准：看完 TL;DR 就知道这篇论文做了什么，不需要再看摘要。**

每条 TL;DR 由"问题"和"方法"两部分组成，每部分都有具体的质量要求：

**问题部分的质量要求（三要素）：**
1. **什么场景/任务**：这篇论文研究的是什么场景
2. **什么痛点**：现有方法/当前实践有什么具体缺陷
3. **为什么不行**：简要说明现有方法为什么解决不了（可选但推荐）

**方法部分的质量要求（三要素）：**
1. **方法名**：提出的方法/框架叫什么
2. **具体怎么做的**：核心技术机制（不只说"用RL"，要说"用RL训练智能体自动执行知识库修正动作"）
3. **关键结果**：如有具体数字或结论，附上（如"8B模型接近专有系统水平"、"NoLiMa上提升10.09"）

### 高质量 TL;DR 示例（✅ 务必参考！）

TL;DR: 问题：视觉语言模型存在"视觉不接地"问题——可能完全靠语言先而非图像回答，现有置信度方法无法检测。方法：提出BICR，在训练时对比真实图像和遮蔽图像的隐藏状态，用排序损失惩罚对遮蔽视图的高置信度，零额外推理成本下实现跨模型最佳校准。

TL;DR: 问题：LLM的隐含偏好不具有文化中立性，现有文化对齐方法需要微调或白盒访问。方法：提出DISCA，推理时将各国建模为人口统计面板，利用分歧而非共识生成有界logit校正，20国7个backbone上文化不对齐降低10-24%，无需改权重。

### 低质量 TL;DR 示例（❌ 严禁！）

```
❌ 问题：研究相关问题。方法：提出新的训练策略。
   → 完全不知道做了什么，零信息量

❌ 问题：研究鲁棒性问题。方法：提出新的注意力机制。
   → 问题和方法都太空泛，什么鲁棒性？什么注意力机制？

❌ 问题：针对现有方法的局限。方法：提出改进方法。
   → 废话，每篇论文都是在改进

❌ 问题：研究精度问题。方法：提出强化学习方法。
   → 缺少具体问题和具体方法描述

❌ 问题：LLM训练存在挑战。方法：提出ELF框架，全面优于基线。
   → 问题太笼统，方法只说框架名不说怎么做

❌ 问题：探索多模态推理方向。方法：取得显著提升。
   → 没说做什么，也没说方法是什么

❌ 问题：RAG系统的检索质量不够好。方法：提出新的检索方法。
   → 比"研究相关问题"好一点，但仍然太空——为什么不好？新方法怎么做？
   → 改进版：问题：RAG系统中检索器与生成器独立优化导致检索结果与生成需求不匹配。方法：提出联合检索-生成训练框架，用生成反馈信号调整检索器，Recall@5提升12%。
```

### TL;DR 撰写步骤（严格按此执行！）

**步骤 1：理解论文**
- 读 title + abstract，在心里用自己的话回答：**这篇论文发现了什么问题？提出了什么方法？方法的核心机制是什么？效果如何？**
- 如果回答不出这四个问题，重读 abstract

**步骤 2：写问题部分**
- 不要写"研究XX问题"——这没有信息量
- 要写：**什么场景下，什么现有方法/当前实践，有什么具体缺陷，为什么不行**
- 好的格式：`[场景/任务]中[现有方法/当前做法]存在[具体缺陷]，因为[原因]`
- 短格式（如果场景明确可省略）：`[具体缺陷]——[根因/为什么现有方法不行]`

**步骤 3：写方法部分**
- 不要写"提出XX方法"然后就没了——这没有信息量
- 要写：**方法名叫什么 + 核心技术机制是什么 + 关键结果/约束（如有）**
- 好的格式：`提出[方法名]，[核心技术机制：具体怎么做的]，[关键结果/约束]`
- 短格式（如果效果不好量化）：`提出[方法名]，[核心技术机制]`

**步骤 4：自检**
- 问自己：**只看这条 TL;DR，我能说出这篇论文具体做了什么吗？**
- 如果答不出→重写
- 如果问题和方法仍然是空泛表述→重写
- 如果方法只说了方法名没说怎么做→补充技术细节

### TL;DR 撰写的禁止做法

1. ❌ **不要翻译完整摘要**——只写 TL;DR 两句话，不翻译 abstract
2. ❌ **不要用脚本/代码自动生成 TL;DR**——脚本只会产出"研究相关问题，提出新方法"
3. ❌ **不要写到单独的文件里**——TL;DR 直接写在最终输出的 `summary.md` 中
4. ❌ **不要用空泛表述**——以下词汇直接表明 TL;DR 质量不合格：
   - "研究相关问题"、"研究XX问题"（没说具体问题是什么）
   - "提出新方法"、"提出改进方法"、"提出新的训练策略"（没说具体怎么做）
   - "取得显著提升"、"全面优于基线"（如果没有数字就没有信息量）
   - "针对现有方法的局限"（每篇论文都是这样，废话）
5. ❌ **不要只说领域不说问题**——"研究LLM训练问题"→ 要说"LLM预训练中上层注意力过早特化导致性能下降"
6. ❌ **不要只说方法类型不说具体内容**——"提出强化学习方法"→ 要说"用RL训练智能体自动执行知识库修正动作"
7. ❌ **不要照搬标题**——标题通常只说领域，TL;DR 要说清问题和方法的细节

## 排除规则

以下情况需在 TL;DR 后追加排除原因标注，但仍**归入对应 topic 并列出**：

| 排除类型 | 追加标注 |
|----------|---------|
| 多语言/小语种/特定语言研究 | `**排除原因：** 多语言/特定语言（xxx 语）` |
| 医学/临床诊断 | `**排除原因：** 医学/临床诊断` |

**判断标准：**

**多语言/特定语言：**
- 标题或摘要提及相关语言研究
- 如"巴西法律文本"、"葡萄牙语"、"低资源语言"、"多语言"、"跨语言"
- 具体语言名（荷兰语、斯瓦希里语、巴西语等）

**医学/临床诊断：**
- 涉及疾病诊断、临床预测、医学影像
- 如"ICU"、"MIMIC"、"临床"、"诊断"、"癌症"、"生存分析"、"药物"、"患者"
- 医学相关数据集或场景

**注意：**
- 排除的论文也要按 topic 分类列出，格式相同
- 排除论文的 TL;DR **仍然要写**（不省略！），只是追加排除标注
- 独作论文不排除，正常写 TL;DR

## 完整流程（按步骤严格执行）

### 阶段一：爬取原始数据

**步骤 1.1** - 创建输出目录（在项目根目录下）：
```bash
mkdir -p {category}_{date}
```

**示例：**
```bash
mkdir -p cs.CL_2026-05-14
```

**步骤 1.2** - 调用爬虫脚本：

**脚本位置：** 项目根目录下的 `papers_cool_crawler.py`

**执行命令（在输出目录下运行）：**
```bash
cd {category}_{date}
python3 ../papers_cool_crawler.py "{URL}"
```

**示例：**
```bash
cd cs.CL_2026-05-14
python3 ../papers_cool_crawler.py "https://papers.cool/arxiv/cs.CL?date=2026-05-14&show=150"
```

**步骤 1.3** - 确认爬虫输出：
- 检查是否生成了 `papers_{category}_{date}.json` 文件
- 如果没有或为空，检查错误信息并重试
- 如果仍然失败，停止并告知用户

**爬虫输出文件：**
- `papers_{category}_{date}.json` - 结构化 JSON（含 title, authors, abstract, subjects, publish_time 等）
- `papers_{category}_{date}.md` - 原始 Markdown（爬虫自动生成）

### 阶段二：读取 TOPICS.md + 通读所有论文 + 分类

**这是最关键的阶段！分类质量决定最终输出的质量。**

**步骤 2.1** - 读取 TOPICS.md：

用 `Read` 工具读取项目根目录下的 `TOPICS.md`，获取当前 topic 候选列表。

**⚠️ 禁止跳过此步！** 不读取 TOPICS.md 就无法复用已有分类，会创建重复 topic。

**步骤 2.2** - 读取 JSON 文件：

用 `Read` 工具读取爬虫生成的 JSON 文件：
```
路径：{category}_{date}/papers_{category}_{date}.json
```

**示例：** `cs.CL_2026-05-14/papers_cs.CL_2026-05-14.json`

JSON 中每篇论文包含字段：`title`, `authors`, `abstract`, `subjects`, `publish_time`, `arxiv_id` 等。

**步骤 2.3** - 通读所有论文的 title + abstract：

**⚠️ 这一步不能省略或跳过！必须完整阅读所有论文后再分类。**

具体操作：
1. 从 JSON 中逐个读取每篇论文的 `title` 和 `abstract`
2. **不要边读边写！** 先完整通读，建立整体印象
3. 通读过程中，可以在心中或草稿中标记论文的大致方向（但不急于确定最终分类）
4. 如果论文数 > 100 篇，可以分批阅读（每批 50 篇），但**必须全部读完再进入步骤 2.4**

**为什么要先通读？**
- 先看全部论文才能判断哪些是同一方向
- 不能仅凭标题分类，有些标题不明显但 abstract 内容明确
- 通读后能更好地发现新 topic

**步骤 2.4** - 对论文进行 topic 分类：

通读完成后，将每篇论文分配到 TOPICS.md 中的 topic 类别。分类规则：
1. **优先使用已有 topic**：检查论文是否匹配 TOPICS.md 中的某个 topic
2. **取最核心的一个**：如果论文涉及多个 topic，选最核心的
3. **创建新 topic**：当论文无法归入任何现有 topic 时：
   - 分配新 Topic ID（小写英文短横线格式，如 `skill-learning`）
   - 写明 Topic 名称和中文说明
   - 在 TOPICS.md 的「🆕 新增 Topic 记录」区域追加新条目
4. **最小粒度**：某 topic 当日只有 1 篇论文时，考虑并入相关 topic

**步骤 2.5** - 为每个 topic 撰写研究动态总结：

针对每个 topic，基于当日论文的 abstract，写一段**该 topic 的研究动态总结**（2-4 句话）：

包含：
- 该 topic **关注什么问题**（核心研究方向）
- **有什么新进展**（方法创新、新发现、新视角）
- **趋势观察**（如果有的话）

要求：
- 不逐篇罗列，**归纳共性、提炼趋势**
- 如果某 topic 只有 1-2 篇论文，总结可简短

### 阶段三：撰写 TL;DR + 输出 Markdown

**⚠️ 这是最容易出错的阶段！请严格按照以下步骤执行，不要跳步或自作主张！**

#### 3.1 TL;DR 撰写（最重要的一步！）

**核心原则：基于阶段二已通读的 abstract，为每篇论文撰写中文 TL;DR。**

**执行方式：**

在阶段三输出 Markdown 时，**每写一篇论文的条目，就同时写它的 TL;DR**。也就是说，TL;DR 不是单独生成再贴进去的，而是**直接写在每篇论文条目中**。

**具体操作（逐 topic 处理）：**

对于每个 topic：
1. 回顾该 topic 下的论文列表和它们的 abstract
2. 逐篇撰写 TL;DR，格式为：`问题：[具体问题] 方法：[具体方法]`
3. 写完一条后，用"自检方法"验证：只看这条 TL;DR，能说出这篇论文具体做了什么吗？
4. 如果不能，重写，直到足够具体

**⚠️ 以下做法严禁：**
- ❌ **不要翻译完整摘要**——如果发现自己在逐句翻译 abstract，立即停止，改为用两句话概括
- ❌ **不要创建单独的 TL;DR 文件**（如 `all_tldr.md`、`all_tldr.txt`）——TL;DR 直接写在最终输出文件中
- ❌ **不要用 Python 脚本批量生成 TL;DR**——脚本生成的 TL;DR 质量极差
- ❌ **不要写空泛 TL;DR**——如"研究相关问题，提出新方法"，必须具体

## 常见错误（务必避免！）

**TL;DR 相关（最容易出错！）：**
- ❌ **翻译完整摘要**——只写 TL;DR，不翻译完整 abstract
- ❌ **用脚本自动生成 TL;DR**——必须基于阅读理解手动撰写
- ❌ **TL;DR 太空泛**——如"研究相关问题，提出新的训练策略"、"研究鲁棒性问题，提出新的注意力机制"
- ❌ **TL;DR 写到单独文件**——TL;DR 直接写在最终输出 `summary.md` 中每篇论文条目里
- ❌ **排除论文不写 TL;DR**——排除论文也要写 TL;DR，只是追加排除原因
- ❌ **TL;DR 只有领域没有问题**——"研究LLM训练问题"不是好 TL;DR
- ❌ **TL;DR 只有类型没有方法**——"提出强化学习方法"不是好 TL;DR

**流程相关：**
- ❌ **不读 TOPICS.md 就分类**——必须先读取现有 topic 列表
- ❌ **不读完所有论文就分类**——必须先通读所有 abstract 再分类
- ❌ **按原始顺序逐篇列出**——必须按 topic 分组
- ❌ **忘记写 topic 研究动态总结**
- ❌ **头部缺少总览表格**
- ❌ **新增 topic 未同步更新 TOPICS.md**
- ❌ **链接格式错误**——应使用 papers.cool 而非 arxiv.org
- ❌ **忘记创建输出目录**
- ❌ **文件名格式错误**——应为 `summary.md`

**格式相关：**
- ❌ **翻译作者名**——作者名保持英文
- ❌ **排除论文忘记写排除原因**
- ❌ **总览表格表头后有空行**——表头行和数据行之间不能有空行，否则 Markdown 表格无法渲染
- ❌ **文件名不用 summary.md**——必须统一为 `summary.md`
- ❌ **保存路径不在 {category}_{date}/ 目录下**——所有文件必须保存在对应的日期目录中

## 经验教训

**为什么写 TL;DR 而不翻译完整摘要：**
- ✅ 效率高：通读所有 abstract 后直接总结，不需要逐句翻译
- ✅ 信息密度高：两句话说清问题和方法，比完整翻译更高效
- ✅ 阅读体验好：用户一眼就能判断论文是否值得深入
- ❌ 翻译完整摘要太慢且信息冗余（用户可以直接看英文 abstract）

**为什么 TL;DR 必须手动撰写而不是脚本生成：**
- ✅ 脚本生成的 TL;DR 质量极差，通常是"研究相关问题，提出新的训练策略"这类空话
- ✅ 只有读完 abstract 并理解后，才能写出具体的、有信息量的 TL;DR
- ✅ 手动撰写可控质量，确保每条 TL;DR 看完就知道论文做了什么

**为什么 TL;DR 直接写在输出文件中而不是单独文件：**
- ✅ 减少中间步骤，避免遗漏或格式不匹配
- ✅ 最终输出就是用户需要的，不要额外合并步骤
- ❌ 写到 `all_tldr.md` 然后合并到 `summary.md`，步骤多且容易出错——TL;DR 直接写在 `summary.md` 中

**为什么按 topic 分组而非原始顺序：**
- ✅ 信息密度高：读者可快速了解当日各方向的研究动态
- ✅ 趋势更明显：同一 topic 的论文放一起，容易发现共性和趋势
- ✅ 阅读效率高：用户可优先关注自己感兴趣的方向

**为什么用 Python 爬虫而不是 browser + evaluate：**
- ✅ 稳定性高：requests 直接获取，不依赖浏览器状态
- ✅ 可复现：给定 URL 一定能爬下来
- ✅ 易维护：Python 脚本清晰，后续好改
- ✅ 离线可用：爬下来的 JSON 可反复处理

**为什么人工分类而非脚本分类：**
- ✅ 语义理解：分类需要理解论文 abstract 的核心贡献
- ✅ 灵活性：可处理跨 topic 论文，自动发现新 topic
- ✅ 质量高：动态总结需要归纳能力，无法自动化

**关键要点（执行前再检查一遍）：**
1. 爬虫脚本保存在同目录下
2. 原始数据保存到 `{category}_{date}/` 子目录中（与 summary.md 同目录）
3. **分类前先读取 `TOPICS.md` 获取最新 topic 列表**
4. **先通读所有论文 abstract，再进行 topic 分类**
5. **为每个 topic 写研究动态总结**
6. **每篇论文写高质量中文 TL;DR，不翻译完整摘要**
7. **TL;DR 质量三要求：问题要说清"什么不行+为什么"，方法要说清"怎么做的"，有数字就带数字**
8. **TL;DR 直接写在最终输出文件 `summary.md` 中，不要写到单独文件**
9. 按 topic 分组输出，每篇论文包含 TL;DR
10. 排除的论文也写 TL;DR，追加排除原因
11. 分批处理：第一批用 `Write`，后续用 `Edit` 追加
12. 输出文件保存到 `{category}_{date}/summary.md`（如 `cs.CL_2026-05-14/summary.md`）
13. **新增 topic 必须同步更新 `TOPICS.md`，并提醒用户审阅**
14. **每写完 10 条 TL;DR，回头检查：是否有"研究相关问题"或"提出新方法"这类空泛表述？有的话立即重写**
